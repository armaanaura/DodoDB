
# Mini Database V1 Requirements Specification

## 1. Document Purpose

This document defines the exact business and engineering requirements for Version 1 of a custom database being built from scratch in Python.

The purpose of Version 1 is:
- to learn core database internals through implementation
- to build a small but correct in-memory database
- to establish a clean architecture that can evolve later
- to avoid over-engineering and unnecessary features in the first version

This version is intended for learning and correctness, not performance or production use.

---

## 2. Version 1 Objective

Version 1 must implement a minimal in-memory database with:
- database management
- table management
- schema definition
- row insertion
- row retrieval
- row update
- row deletion
- schema validation
- predictable error handling
- automated tests

Version 1 must be complete enough to demonstrate a functioning mini database with CRUD behavior and schema enforcement.

---

## 3. In Scope

The following capabilities are required in Version 1.

### 3.1 Database-Level Features
- create a database object
- create tables
- retrieve a table by name
- list existing tables
- check whether a table exists

### 3.2 Table-Level Features
- create columns before inserting rows
- store rows in memory
- insert one row
- insert multiple rows
- retrieve all rows
- retrieve rows matching equality conditions
- update rows matching equality conditions
- delete rows matching equality conditions
- count rows

### 3.3 Schema-Level Features
- each table must have a defined schema
- each column must have:
  - name
  - data type
  - nullability rule
- duplicate column names must be rejected
- invalid column types must be rejected
- rows must be validated against schema before insertion
- updated rows must also be validated against schema before being saved

### 3.4 Error Handling
- invalid operations must raise clear exceptions
- silent failures are not allowed
- invalid data must not be inserted

### 3.5 Testing
- automated unit tests must exist for core behaviors
- tests must verify correctness of schema rules and CRUD operations

---

### 4.1 Query Language

## 5. Functional Requirements

## FR-1: Database Creation
The system shall allow creation of a database object.

### Acceptance Criteria
- a database object can be instantiated with a name
- the database object maintains a collection of tables

---

## FR-2: Table Creation
The system shall allow creation of a table inside the database.

### Acceptance Criteria
- a table can be created by name
- duplicate table names are rejected
- table objects are retrievable from the database

---

## FR-3: Column Creation
The system shall allow columns to be created inside a table before row insertion begins.

### Acceptance Criteria
- a column has a name, type, and nullable flag
- duplicate column names in the same table are rejected
- invalid data types are rejected
- schema must remain consistent after definition

---

## FR-4: Row Insertion
The system shall allow insertion of rows into a table.

### Acceptance Criteria
- inserted rows must use valid column names only
- inserted values must match the declared column types
- missing non-nullable columns are rejected
- missing nullable columns are filled with `None`
- inserted rows are copied before storage to avoid external mutation
- invalid rows are rejected before insertion

---

## FR-5: Retrieve All Rows
The system shall allow retrieval of all rows from a table.

### Acceptance Criteria
- all inserted rows are returned
- returned rows are copies, not internal mutable references
- retrieval does not mutate stored data

---

## FR-6: Equality Filtering
The system shall allow retrieval of rows matching equality-based conditions.

### Acceptance Criteria
- filtering supports one or more column-value conditions
- a row matches only if all provided conditions match
- invalid filter columns are rejected or treated consistently by design
- results are returned as row copies

---

## FR-7: Row Update
The system shall allow updating rows that match equality-based conditions.

### Acceptance Criteria
- only matching rows are updated
- updated row values must be schema-validated before commit
- invalid updates are rejected
- method returns the number of updated rows

---

## FR-8: Row Deletion
The system shall allow deletion of rows that match equality-based conditions.

### Acceptance Criteria
- only matching rows are deleted
- method returns the number of deleted rows
- non-matching rows remain unchanged

---

## FR-9: Table Listing
The system shall allow listing tables in the database.

### Acceptance Criteria
- table names can be retrieved as a list
- result format must be predictable

---

## FR-10: Column Listing
The system shall allow listing columns in a table.

### Acceptance Criteria
- column names can be retrieved as a list
- optional detailed output may include type and nullability
- core API should return values, not only print them

---

## FR-11: Row Count
The system shall provide row count for a table.

### Acceptance Criteria
- row count reflects current number of stored rows
- row count changes correctly after insert, update, and delete operations

---

## 6. Non-Functional Requirements

## NFR-1: Simplicity
The implementation must prioritize simplicity and clarity over performance.

## NFR-2: Correctness
Schema validation and CRUD behavior must be correct before any optimization work begins.

## NFR-3: Small Scope
Version 1 must remain intentionally small and understandable.

## NFR-4: Maintainability
Code must be organized into clear modules with separated responsibilities.

## NFR-5: Testability
Core logic must be covered by automated tests.

## NFR-6: Predictability
Public methods must have stable, explicit behavior and clear errors.

## 7. Recommended Internal Data Model

## 7.1 Column Representation
A column shall contain:
- `name`
- `dtype`
- `nullable`

Example:
```python
Column("id", int, nullable=False)
````

## 7.2 Schema Representation

A schema shall be represented as:

* ordered list of columns
* internal dictionary for fast lookup by column name

Example:

```python
[
    Column("id", int, nullable=False),
    Column("name", str, nullable=False),
    Column("age", int, nullable=True),
]
```

## 7.3 Row Representation

Rows shall be represented as:

```python
dict[str, Any]
```

Example:

```python
{"id": 1, "name": "Arman", "age": 24}
```

## 7.4 Table Representation

A table shall maintain:

* `name`
* `schema`
* `rows: list[dict[str, Any]]`

## 7.5 Database Representation

A database shall maintain:

* `tables: dict[str, Table]`

---

## 8. Required Class Design

Version 1 shall include the following core classes.

### 8.1 Column

Responsibility:

* represent one column definition

Required fields:

* `name`
* `dtype`
* `nullable`

### 8.2 Schema

Responsibility:

* validate schema definition
* validate rows
* normalize rows before storage

Required behaviors:

* reject duplicate columns
* reject empty schema if chosen by design
* validate row correctness
* fill missing nullable columns with `None`

### 8.3 Table

Responsibility:

* hold rows
* enforce schema through schema object
* implement CRUD row operations

### 8.4 Database

Responsibility:

* manage tables
* create tables
* retrieve tables
* list tables

### 8.5 Exceptions

Responsibility:

* provide meaningful error types for invalid operations

---

## 9. Required Public API

The following method set is required for Version 1.

## 9.1 Column

```python
class Column:
    def __init__(self, column_name: str, column_dtype: type, nullable: bool = True): ...
```

## 9.2 Schema

```python
class Schema:
    def __init__(self, columns: list[Column]) -> None: ...
    def validate_definition(self) -> None: ...
    def validate_row(self, row: dict[str, Any]) -> None: ...
    def normalize_row(self, row: dict[str, Any]) -> dict[str, Any]: ...
    def column_names(self) -> list[str]: ...
```

## 9.3 Table

```python
class Table:
    def __init__(self, table_name: str, schema: Schema) -> None: ...
    def insert(self, row: dict[str, Any]) -> None: ...
    def insert_many(self, rows: list[dict[str, Any]]) -> None: ...
    def select_all(self) -> list[dict[str, Any]]: ...
    def select_where(self, where: dict[str, Any] | None = None) -> list[dict[str, Any]]: ...
    def update_where(self, where: dict[str, Any], values: dict[str, Any]) -> int: ...
    def delete_where(self, where: dict[str, Any]) -> int: ...
    def row_count(self) -> int: ...
    def list_columns(self, detailed: bool = False): ...
```

## 9.4 Database

```python
class Database:
    def __init__(self, database_name: str, metadata: str = "") -> None: ...
    def create_table(self, table_name: str, columns: list[Column]) -> Table: ...
    def get_table(self, table_name: str) -> Table: ...
    def list_tables(self) -> list[str]: ...
    def has_table(self, table_name: str) -> bool: ...
```

## 10. Error Handling Requirements

Version 1 must define custom exceptions.

Required exception hierarchy:

```python
class DatabaseError(Exception): ...
class TableAlreadyExistsError(DatabaseError): ...
class TableNotFoundError(DatabaseError): ...
class ColumnAlreadyExistsError(DatabaseError): ...
class SchemaError(DatabaseError): ...
class ValidationError(DatabaseError): ...
```

### Error Handling Rules

* invalid user input must raise a clear exception
* missing tables must raise a table-not-found error
* duplicate tables must raise a duplicate-table error
* invalid rows must raise a validation error
* duplicate columns must raise a schema or column error

---

## 11. State Management Rules

The following rules must hold true in Version 1.

### 11.1 Row Immutability from Outside

* rows returned by the table must be copies
* caller mutation must not change internal stored data

### 11.2 Schema Mutation Restriction

* schema changes after row insertion should be disallowed in Version 1
* if schema mutation is allowed, all old rows must be backfilled consistently
* recommended decision for V1: disallow schema changes after the first inserted row

### 11.3 Column Rename Restriction

* direct column renaming must not be supported in V1
* renaming a column without updating table metadata and stored rows causes inconsistency
* recommendation: do not implement `change_name()` in V1

## 13. Testing Requirements

Version 1 shall include automated tests using `pytest`.

## 13.1 Schema Tests

Required tests:

* schema rejects duplicate column names
* schema rejects empty column list if that rule is enforced
* valid row passes validation
* unknown column is rejected
* wrong type is rejected
* missing non-nullable column is rejected
* missing nullable column is normalized to `None`

## 13.2 Database Tests

Required tests:

* create table succeeds
* duplicate table creation fails
* get existing table succeeds
* get missing table fails
* list tables returns expected names

## 13.3 Table Tests

Required tests:

* insert one valid row
* insert many rows
* select all returns all rows
* select all returns copies
* select where works with one condition
* select where works with multiple conditions
* update modifies matching rows only
* update returns updated count
* invalid update is rejected
* delete removes matching rows only
* delete returns deleted count
* row count is correct

## 14. Acceptance Criteria for Version 1 Completion

Version 1 shall be considered complete only when all of the following are true:

1. multiple tables can be created in one database
2. schemas are enforced during insert and update
3. nullable and non-nullable behavior works correctly
4. row insertion works
5. full row scan works
6. equality filtering works
7. row update works
8. row delete works
9. invalid operations raise custom exceptions
10. automated tests pass
11. a demo script works end-to-end without manual fixes
12. code remains small, understandable, and not over-engineered

## 15. Demo Requirement

The first successful Version 1 demo must do the following:

1. create a database
2. create a `users` table
3. define columns:

   * `id: int, nullable=False`
   * `name: str, nullable=False`
   * `age: int, nullable=True`
4. insert at least three rows
5. retrieve all rows
6. filter rows by one field
7. update one row
8. delete one row
9. print final remaining rows
10. demonstrate one invalid insert being rejected by schema validation

Example demo scenario:

* insert 3 users
* filter where `name == "Arman"`
* update age where `id == 2`
* delete row where `id == 3`
* attempt invalid insert with wrong type for `id`

## 16. Risks to Avoid

The following are common failure modes and must be avoided:

* starting SQL parsing before storage behavior is complete
* adding indexes before table scan works correctly
* allowing partial invalid updates
* exposing internal row references directly
* mixing schema logic into too many unrelated places
* over-designing abstractions for future storage engines
* building printing-based APIs instead of return-value-based APIs
* allowing uncontrolled schema mutation after data insertion
* skipping tests because the code feels simple

---

## 17. Recommended Implementation Order

The implementation must follow this order:

### Phase 1

* exceptions
* column model
* schema class
* database skeleton

### Phase 2

* table creation with schema
* insert
* insert_many
* select_all
* row_count

### Phase 3

* equality matching helper
* select_where
* update_where
* delete_where

### Phase 4

* test suite
* demo script
* cleanup and refactor

No other phases should be started before these are complete.

---

## 18. Version 1 Exit Decision

If all requirements in this document are satisfied, Version 1 is complete.

Once Version 1 is complete, the recommended next step is:

### Version 2

Add persistence to disk.

That should come before:

* SQL parser
* indexing
* transaction handling
* optimizer work

Reason:
Persistence is the next real database concern after correct in-memory behavior.
